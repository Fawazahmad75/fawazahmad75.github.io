That middle is where the real engineering lives. It's where you choose a classifier that catches 82% of negative reviews over one with a higher headline F1 because the business cost of a missed complaint outweighs a false flag. It's where you add a confidence threshold to route uncertain predictions to humans. It's where you pair dense embeddings with BM25 and fuse them with Reciprocal Rank Fusion because retrieval problems rarely have one right shape.

I also work the floor at Walmart — stocking shelves, occasionally covering customer-facing shifts — and it's where my interest in retention analytics actually started. Watching which customers come back, which complaints get ignored, which products vanish, you start to see the shape of data that a dashboard alone will never capture. That observation became my first real production ML project.

I'm looking for a role where I can keep learning this way: real problems, messy data, experienced mentors, and room to own the trade-offs.

E-commerce and retail businesses lose a significant portion of revenue to silent churn — customers who stop buying without ever complaining. The raw signals (declining orders, rising complaints, stale engagement) are buried in transactional data, and written feedback compounds the problem: negative reviews often get averaged out by a strong headline sentiment score.

The goal: build a single intelligence layer that predicts who will churn, explains why, segments customers by value, and flags negative feedback before it becomes a cancellation.

Two parallel pipelines, unified into one dashboard:

  • Structured side (churn): Trained four models on a 5,630-customer dataset — Logistic Regression, Gradient Boosting, Random Forest, and a custom PyTorch neural network with a hand-built Dataset class, training loop, and dropout regularization. Random Forest won on F1 and calibration.
  • Unstructured side (NLP): Built a TF-IDF + Logistic Regression sentiment classifier on 22,641 customer reviews, with a confidence threshold routing uncertain predictions to human review.
  • Business layer: Added RFM segmentation (Champions, Loyal, At Risk, New, Lost, Needs Attention) and CLV estimation so predictions become actions.

Modeling: Python, scikit-learn, PyTorch (custom training loop + Dataset class).

Explainability: SHAP for global and per-customer feature attribution.

Data: pandas, NumPy for cleaning, encoding, imputation, and RFM feature engineering.

Deployment: Streamlit frontend, deployed to Streamlit Cloud via GitHub.

Random Forest — 98% test accuracy, 0.95 F1 on churn prediction.

SHAP analysis surfaced the top three churn drivers: tenure, complaint history, and days since last order — matching qualitative observations from retail work and giving the dashboard real explanatory power.

Deliberate metric trade-off: For sentiment classification, chose a threshold-based model that recalls 82% of negative reviews (vs. 33% for a higher-F1 vanilla Random Forest). The business cost of a missed complaint outweighs a false flag — a decision I'd defend in any stakeholder room.

Shipped: Unified Streamlit dashboard combining churn risk, RFM segment, CLV estimate, sentiment, and context-aware retention recommendations into a single report per customer.

System Architecture
i. context

The Problem

Subscription and e-commerce businesses lose 15–30% of customers annually. Most retention systems fail because they treat all churn as equally bad (losing a $12 customer doesn't justify a $20 retention offer), rely on structured data alone, provide predictions without explanations, and don't differentiate high-value loyalists from low-value drifters. This project addresses all four gaps in one unified system.

Only 16.8% of customers in the dataset churned. A naive "predict no churn" model would score 83% accuracy while catching zero churners — which is why F1 and churner recall, not headline accuracy, drove model selection. The default Random Forest was chosen over the tuned version because the tuned model dropped recall from 91% to 67% to gain a 1.3% reduction in overfitting gap. In a context where missing a churner is more costly than flagging a stable customer, that trade is wrong.

The PyTorch network was included to empirically validate the choice of Random Forest, not because deep learning was expected to win. On small tabular datasets with strong feature signals, tree ensembles outperform neural networks because they natively handle mixed feature types and capture non-linear thresholds without requiring scaling. Training loss decreased smoothly from 0.24 to 0.097 across 100 epochs — a healthy curve, just not a winning model.

v. explainability

SHAP Analysis

SHAP transformed the Random Forest from a black box into an auditable system. The top five churn drivers were tenure, complaints, days since last order, single marital status, and number of saved addresses — the last of which was non-obvious and possibly indicates customers shopping across multiple shipping contexts (gifting, multiple residences) being less stably engaged. SHAP and built-in feature importance were both included because they measure different things: built-in importance tracks usage frequency in tree splits; SHAP measures actual contribution to individual predictions using Shapley values from cooperative game theory.

viii. unified intelligence

The Full Analysis Tab

The deployed app's killer feature is the Full Analysis tab, which combines behavioral data and review text into a single intelligence report. It handles four distinct scenarios:

  • High churn risk + Negative feedback → Critical: immediate intervention, root-cause complaint resolution
  • High churn risk + Positive feedback → Quiet disengagement: customer is leaving without complaining; proactive outreach
  • Low churn risk + Negative feedback → Vocal complaint, stable customer: quick fix preserves the relationship
  • Low churn risk + Positive feedback → Healthy: maintain engagement, candidate for referral programs

Most retention dashboards stop at the prediction. This one acts on the combined signal.

Vanilla RAG systems have two well-known failure modes: they either over-rely on semantic similarity (and miss exact keyword matches like product codes, acronyms, or proper nouns), or they hallucinate answers when retrieval returns nothing useful and the LLM decides to improvise.

The goal: build a production-ready document Q&A system that handles both failure modes — with grounded, citation-enforced answers and sub-second latency.

A full pipeline, designed for explicit failure handling:

  • Ingestion: Parse PDF/text, chunk into 500-word segments with 50-word overlap to preserve context at chunk boundaries.
  • Embedding + indexing: sentence-transformers (all-MiniLM-L6-v2) for dense vectors stored in ChromaDB.
  • Hybrid retrieval: Dense search + BM25 keyword search, fused via Reciprocal Rank Fusion (RRF) — dense handles semantic recall, BM25 anchors exact-match precision.
  • Generation: Llama 3.1 via Groq API, wrapped in a structured prompt that constrains answers to retrieved context and requires inline source citations.
  • Serving: FastAPI backend deployed on Railway, Streamlit frontend for upload + Q&A.

Embeddings: sentence-transformers (all-MiniLM-L6-v2).

Vector store: ChromaDB.

Keyword search: BM25 (rank-bm25).

LLM: Llama 3.1 via Groq API.

Serving: FastAPI backend, Streamlit frontend, deployed on Railway.

Retrieval latency consistently under 50ms per query; end-to-end (retrieval + generation) under 1 second on Groq.

Hybrid retrieval noticeably outperformed pure dense search on queries containing exact identifiers, acronyms, and proper nouns — the class of queries where semantic embeddings silently fail.

Hallucination control: Structured prompt + citation requirement eliminated ungrounded generation in testing. Answers either cite a chunk or decline.

Deployed as a FastAPI service with latency instrumentation per query, ready to plug into a real product workflow.

Pipeline Architecture
i. ingestion

Chunking Strategy

Documents are parsed with PyMuPDF (fitz), then chunked into 500-word segments with a 50-word overlap. Chunk size balances retrieval precision (smaller = more focused matches) against context richness (larger = better LLM grounding). The 50-word overlap exists so that a question whose answer straddles a chunk boundary doesn't get cut in half by the chunker — every concept appears in full in at least one chunk.

ii. dense retrieval

Embeddings & Vector Store

Each chunk is encoded into a 384-dimensional vector by sentence-transformers (all-MiniLM-L6-v2) and stored in ChromaDB. all-MiniLM-L6-v2 was chosen for its speed-quality balance — large enough to capture meaningful semantics, small enough to embed thousands of chunks in seconds without GPU. ChromaDB runs as an EphemeralClient on the deployed Railway instance because users upload their own documents per session and persistence isn't needed; this also avoids filesystem permission issues on the hosting environment.

iii. fusion

Hybrid Retrieval & RRF

Pure semantic search misses queries that hinge on exact tokens — product codes, acronyms, proper nouns. Pure BM25 keyword search misses paraphrased queries. The system runs both retrievers in parallel and fuses their ranked lists using Reciprocal Rank Fusion:

score(d) = Σ  1 / (k + rank_i(d))
           i

k is set to 60 by convention. RRF requires no score calibration between the two retrievers — it operates purely on ranks — which makes it robust when combining methods that produce scores on entirely different scales (cosine similarity vs BM25 term-weighting).

iv. hallucination control

Prompt Engineering

You are a document Q&A assistant. Answer ONLY using the
provided context. Cite source chunks with [1], [2], etc.
If the context does not contain the answer, say:
"I cannot answer this from the provided documents."
Do not use prior knowledge.

CONTEXT:
{retrieved_chunks}

QUESTION: {user_query}

Three constraints in one prompt: context-only, mandatory inline citations, and explicit decline-over-improvise. In testing this eliminated ungrounded generation — the model either cites or declines.

v. production

Latency & Deployment

Per-stage latency on the deployed Railway instance:

StageLatency
Embedding query~10ms
Hybrid retrieval (dense + BM25 + RRF)20–45ms
LLM generation (Llama 3.1 8B via Groq)500–1000ms
End-to-endtypically under 1 second

Groq was chosen for hosted Llama 3.1 inference because of fast token throughput and a generous free tier — the system stays demo-ready without a local GPU. FastAPI exposes /upload, /ask, and /stats endpoints; Streamlit provides the upload + Q&A frontend. The backend is publicly accessible at /docs for API exploration.
